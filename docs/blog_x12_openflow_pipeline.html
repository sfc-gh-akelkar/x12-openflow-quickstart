From 300MB EDI Files to AI-Enriched Claims in 15 Minutes

Building a production X12 pipeline on Snowflake — serverless ingestion, declarative parsing, and Claude Sonnet enrichment with zero orchestration code.

Openflow (NiFi on SPCS) Snowpipe Streaming Dynamic Tables Cortex AI Claude Sonnet 4-6

Healthcare organizations receive hundreds of megabytes of X12 EDI files from clearinghouses every day. These files contain claims, enrollments, and remittance data encoded in a format designed in the 1970s — no schema, no JSON, no CSV headers. Just pipe-delimited segments separated by tildes.

The legacy approach: custom parsers, ETL jobs, staging databases, hours of delay. The modern approach: a file lands in S3, streams into Snowflake without a warehouse, gets parsed by declarative Dynamic Tables, and is enriched by Claude Sonnet — all within 15 minutes, zero orchestration code.

This post walks through the full architecture, from raw bytes to AI-enriched analytics.

What is X12 EDI?

X12 is the electronic data interchange standard used across US healthcare for transmitting claims (837), enrollments (834), remittance advice (835), eligibility (270/271), and more. Every payer, provider, and clearinghouse speaks X12.

Here's what a single professional claim (837P) looks like — one patient, one visit, three service lines:

NM1*41*2*TCH URGENT CARE KATY*****46*1822835714~PER*IC*BILLING DEPT*TE*7135551234~
NM1*40*2*UNITED HEALTHCARE*****46*86111~HL*1**20*1~
NM1*85*2*TCH URGENT CARE KATY*****XX*1822835714~N3*6621 FANNIN ST~N4*HOUSTON*TX*77030~
REF*EI*741852963~HL*2*1*22*1~
NM1*IL*1*SMITH*JAMES****MI*TCH849459~N3*3983 MAIN ST~N4*HOUSTON*TX*77030~
DMG*D8*20100501*M~HL*3*2*23*0~

SV1*HC~99213*398.43*UN*1***1~DTP*472*D8*20250706~REF*6R*LN001~
SV1*HC~73610*379.51*UN*1***1~DTP*472*D8*20250706~REF*6R*LN002~

The anatomy:

ST/SETransaction envelope (start/end) NM1Name segments (patient, provider, payer — differentiated by qualifier in position 2) CLMClaim header (ID, billed amount, place of service) SV1Service line (procedure code, charge) HIDiagnosis (ICD-10 code) DMGDemographics (DOB, gender)

Fields are positional, separated by *. Segments are terminated by ~. There are no column headers. A 300MB file contains 500K+ of these transactions concatenated together.

Step 1: Openflow Flow Configuration

Create an Openflow deployment and runtime, then build this 6-processor flow:

ListS3 → FetchS3Object → SplitContent → ExtractText → ReplaceText → PutSnowpipeStreaming

ListS3 — Bucket: your S3 bucket, Prefix: x12/, Listing Strategy: Tracking Timestamps

FetchS3Object — Same bucket/credentials as ListS3

SplitContent — Byte Sequence: ST*, Keep Byte Sequence: Leading. This splits a 300MB file into individual transactions at transaction set boundaries.

ExtractText — Dynamic property: x12_content = ([\s\S]+). Captures the entire FlowFile content as an attribute.

ReplaceText — Replacement Strategy: Always Replace, Evaluation Mode: Entire text

{"RAW_X12_DATA":"${x12_content:escapeJson()}","SOURCE_SYSTEM":"OPENFLOW_S3","MESSAGE_ID":"${UUID()}","INGESTION_TIMESTAMP":${now():toNumber()}}

PutSnowpipeStreaming (v1) — Account locator, Key Pair auth, Database: X12_EDI_AI, Schema: STAGING, Table: LANDING_X12_RAW, Record Reader: JsonTreeReader, Client Lag: 1 sec

Why PutSnowpipeStreaming v1 over v2?

v1 uses a Record Reader and targets a Table directly — perfect for structured JSON records. v2 uses NDJSON + File Fragments targeting a Pipe with Offset Tokens — designed for exactly-once file-level delivery. For streaming individual parsed records, v1 is the right choice.

Step 2: Bronze Landing Table

CREATE DATABASE IF NOT EXISTS X12_EDI_AI;
CREATE SCHEMA IF NOT EXISTS X12_EDI_AI.STAGING;

CREATE OR REPLACE TABLE X12_EDI_AI.STAGING.LANDING_X12_RAW (
    MESSAGE_ID          VARCHAR(100),
    TRANSACTION_TYPE    VARCHAR(10),
    RAW_X12_DATA        VARCHAR(16777216) NOT NULL,
    SOURCE_SYSTEM       VARCHAR(100),
    INGESTION_TIMESTAMP VARCHAR(100)
);

Important: Snowpipe Streaming does not support columns with DEFAULT values, IDENTITY, AUTOINCREMENT, or TIMESTAMP types with format expectations. Keep all columns as plain VARCHAR — cast downstream.

Step 3: Silver Layer — Parsing X12 with Dynamic Tables

This is where the magic happens. We turn opaque EDI strings into structured, queryable tables using pure SQL.

The Core Insight: SPLIT_TO_TABLE

X12 uses ~ as the segment terminator. One transaction with 25 segments becomes 25 rows:

CREATE OR REPLACE DYNAMIC TABLE X12_EDI_AI.SILVER.SILVER_X12_SEGMENTS
    TARGET_LAG = '5 minutes'
    WAREHOUSE = APP_WH
AS
SELECT
    r.MESSAGE_ID,
    r.SOURCE_SYSTEM,
    r.INGESTION_TIMESTAMP,
    seg.INDEX AS SEGMENT_POSITION,
    TRIM(seg.VALUE) AS RAW_SEGMENT,
    SPLIT_PART(TRIM(seg.VALUE), '*', 1) AS SEGMENT_ID,
    TRIM(seg.VALUE) AS SEGMENT_DATA
FROM X12_EDI_AI.STAGING.LANDING_X12_RAW r,
    LATERAL SPLIT_TO_TABLE(r.RAW_X12_DATA, '~') seg
WHERE TRIM(seg.VALUE) != '';

4.4M transactions → 99M segment rows. Each row now has a SEGMENT_ID and positional SEGMENT_DATA we can parse with SPLIT_PART.

Claims Header

CREATE OR REPLACE DYNAMIC TABLE X12_EDI_AI.SILVER.SILVER_CLAIMS_HEADER
    TARGET_LAG = '5 minutes'
    WAREHOUSE = APP_WH
AS
SELECT
    MESSAGE_ID,
    SPLIT_PART(SEGMENT_DATA, '*', 2) AS CLAIM_ID,
    TRY_TO_DECIMAL(SPLIT_PART(SEGMENT_DATA, '*', 3), 12, 2) AS TOTAL_BILLED_AMOUNT,
    SPLIT_PART(SEGMENT_DATA, '*', 6) AS PLACE_OF_SERVICE_CODE,
    SOURCE_SYSTEM,
    INGESTION_TIMESTAMP
FROM X12_EDI_AI.SILVER.SILVER_X12_SEGMENTS
WHERE SEGMENT_ID = 'CLM';

Patients

CREATE OR REPLACE DYNAMIC TABLE X12_EDI_AI.SILVER.SILVER_PATIENTS
    TARGET_LAG = '5 minutes'
    WAREHOUSE = APP_WH
AS
WITH patient_nm1 AS (
    SELECT MESSAGE_ID,
        SPLIT_PART(SEGMENT_DATA, '*', 4) AS PATIENT_LAST_NAME,
        SPLIT_PART(SEGMENT_DATA, '*', 5) AS PATIENT_FIRST_NAME,
        SPLIT_PART(SEGMENT_DATA, '*', 10) AS MEMBER_ID,
        SEGMENT_POSITION
    FROM X12_EDI_AI.SILVER.SILVER_X12_SEGMENTS
    WHERE SEGMENT_ID = 'NM1' AND SPLIT_PART(SEGMENT_DATA, '*', 2) = 'IL'
),
patient_dmg AS (
    SELECT MESSAGE_ID,
        SPLIT_PART(SEGMENT_DATA, '*', 3) AS DATE_OF_BIRTH,
        SPLIT_PART(SEGMENT_DATA, '*', 4) AS GENDER_CODE
    FROM X12_EDI_AI.SILVER.SILVER_X12_SEGMENTS
    WHERE SEGMENT_ID = 'DMG'
)
SELECT
    n.MESSAGE_ID,
    n.PATIENT_LAST_NAME,
    n.PATIENT_FIRST_NAME,
    n.MEMBER_ID,
    TRY_TO_DATE(d.DATE_OF_BIRTH, 'YYYYMMDD') AS DATE_OF_BIRTH,
    CASE d.GENDER_CODE WHEN 'M' THEN 'Male' WHEN 'F' THEN 'Female' ELSE 'Unknown' END AS GENDER
FROM patient_nm1 n
LEFT JOIN patient_dmg d ON n.MESSAGE_ID = d.MESSAGE_ID;

Providers

CREATE OR REPLACE DYNAMIC TABLE X12_EDI_AI.SILVER.SILVER_PROVIDERS
    TARGET_LAG = '5 minutes'
    WAREHOUSE = APP_WH
AS
SELECT
    MESSAGE_ID,
    CASE SPLIT_PART(SEGMENT_DATA, '*', 2)
        WHEN '85' THEN 'Billing'
        WHEN '82' THEN 'Rendering'
        WHEN '41' THEN 'Submitter'
    END AS PROVIDER_ROLE,
    SPLIT_PART(SEGMENT_DATA, '*', 4) AS PROVIDER_NAME,
    SPLIT_PART(SEGMENT_DATA, '*', 10) AS PROVIDER_NPI
FROM X12_EDI_AI.SILVER.SILVER_X12_SEGMENTS
WHERE SEGMENT_ID = 'NM1' AND SPLIT_PART(SEGMENT_DATA, '*', 2) IN ('85', '82', '41');

Diagnoses — The Tilde Problem

Here's the gotcha. In X12, ~ is used as both the segment terminator AND a component separator within certain segments. The diagnosis segment HI*ABK~K21.0 contains a ~ between the qualifier (ABK) and the ICD-10 code (K21.0). When we split the entire transaction on ~, these get separated into adjacent rows.

The fix: JOIN to the next SEGMENT_POSITION:

CREATE OR REPLACE DYNAMIC TABLE X12_EDI_AI.SILVER.SILVER_DIAGNOSES
    TARGET_LAG = '5 minutes'
    WAREHOUSE = APP_WH
AS
SELECT
    hi.MESSAGE_ID,
    icd.RAW_SEGMENT AS ICD10_CODE,
    'Principal' AS DIAGNOSIS_TYPE
FROM X12_EDI_AI.SILVER.SILVER_X12_SEGMENTS hi
JOIN X12_EDI_AI.SILVER.SILVER_X12_SEGMENTS icd
    ON hi.MESSAGE_ID = icd.MESSAGE_ID
    AND icd.SEGMENT_POSITION = hi.SEGMENT_POSITION + 1
WHERE hi.SEGMENT_ID = 'HI';

The same pattern applies to SV1*HC~99213 — the procedure code is in the row after the SV1 segment:

CREATE OR REPLACE DYNAMIC TABLE X12_EDI_AI.SILVER.SILVER_SERVICE_LINES
    TARGET_LAG = '5 minutes'
    WAREHOUSE = APP_WH
AS
SELECT
    sv.MESSAGE_ID,
    sv.SEGMENT_POSITION,
    SPLIT_PART(sld.RAW_SEGMENT, '*', 1) AS PROCEDURE_CODE,
    TRY_TO_DECIMAL(SPLIT_PART(sld.RAW_SEGMENT, '*', 2), 12, 2) AS LINE_CHARGE_AMOUNT,
    SPLIT_PART(sld.RAW_SEGMENT, '*', 3) AS UNIT_TYPE,
    TRY_TO_NUMBER(SPLIT_PART(sld.RAW_SEGMENT, '*', 4)) AS UNIT_COUNT,
    sv.SOURCE_SYSTEM,
    sv.INGESTION_TIMESTAMP
FROM X12_EDI_AI.SILVER.SILVER_X12_SEGMENTS sv
JOIN X12_EDI_AI.SILVER.SILVER_X12_SEGMENTS sld
    ON sv.MESSAGE_ID = sld.MESSAGE_ID
    AND sld.SEGMENT_POSITION = sv.SEGMENT_POSITION + 1
WHERE sv.SEGMENT_ID = 'SV1';

Step 4: Gold Layer — Analytics-Ready + AI Enrichment

Unified Claims Table

Join all Silver tables into a single analytics-ready record:

CREATE OR REPLACE DYNAMIC TABLE X12_EDI_AI.GOLD.GOLD_CLAIMS_COMPLETE
    TARGET_LAG = '10 minutes'
    WAREHOUSE = APP_WH
AS
SELECT
    c.MESSAGE_ID, c.CLAIM_ID, c.TOTAL_BILLED_AMOUNT, c.PLACE_OF_SERVICE_CODE,
    p.PATIENT_LAST_NAME, p.PATIENT_FIRST_NAME, p.MEMBER_ID, p.DATE_OF_BIRTH, p.GENDER,
    bp.PROVIDER_NAME AS BILLING_PROVIDER_NAME, bp.PROVIDER_NPI AS BILLING_PROVIDER_NPI,
    rp.PROVIDER_NAME AS RENDERING_PROVIDER_NAME,
    d.ICD10_CODE AS PRIMARY_DIAGNOSIS_CODE,
    COUNT(DISTINCT sl.SEGMENT_POSITION) AS SERVICE_LINE_COUNT,
    c.SOURCE_SYSTEM, c.INGESTION_TIMESTAMP
FROM X12_EDI_AI.SILVER.SILVER_CLAIMS_HEADER c
LEFT JOIN X12_EDI_AI.SILVER.SILVER_PATIENTS p ON c.MESSAGE_ID = p.MESSAGE_ID
LEFT JOIN X12_EDI_AI.SILVER.SILVER_PROVIDERS bp ON c.MESSAGE_ID = bp.MESSAGE_ID AND bp.PROVIDER_ROLE = 'Billing'
LEFT JOIN X12_EDI_AI.SILVER.SILVER_PROVIDERS rp ON c.MESSAGE_ID = rp.MESSAGE_ID AND rp.PROVIDER_ROLE = 'Rendering'
LEFT JOIN X12_EDI_AI.SILVER.SILVER_DIAGNOSES d ON c.MESSAGE_ID = d.MESSAGE_ID
LEFT JOIN X12_EDI_AI.SILVER.SILVER_SERVICE_LINES sl ON c.MESSAGE_ID = sl.MESSAGE_ID
GROUP BY ALL;

AI Enrichment with Claude Sonnet

ICD-10 codes like K21.0 or J45.20 are meaningless to business analysts. We use Snowflake's AI_COMPLETE function with Claude Sonnet to decode them:

CREATE OR REPLACE DYNAMIC TABLE X12_EDI_AI.GOLD.GOLD_CLAIMS_AI_ENRICHED
    TARGET_LAG = '30 minutes'
    WAREHOUSE = APP_WH
AS
WITH enrichment AS (
    SELECT gc.*,
        TRY_PARSE_JSON(
            AI_COMPLETE('claude-sonnet-4-6',
                'You are a medical coding expert. Given this ICD-10 code, return a JSON object '
                || 'with keys: "description", "category" (Respiratory, Cardiovascular, '
                || 'Musculoskeletal, Endocrine, Gastrointestinal, Infectious, Neoplasm, Injury, '
                || 'Mental Health, Other), "chronic" (true/false). Return ONLY valid JSON. '
                || 'ICD-10 code: ' || COALESCE(gc.PRIMARY_DIAGNOSIS_CODE, 'Z00.00')
            )
        ) AS AI_ENRICHMENT
    FROM X12_EDI_AI.GOLD.GOLD_CLAIMS_COMPLETE gc
)
SELECT *,
    AI_ENRICHMENT:description::VARCHAR AS DIAGNOSIS_DESCRIPTION,
    AI_ENRICHMENT:category::VARCHAR AS DIAGNOSIS_CATEGORY,
    AI_ENRICHMENT:chronic::BOOLEAN AS IS_CHRONIC_CONDITION
FROM enrichment;

SELECT
    DIAGNOSIS_CATEGORY,
    COUNT(*) AS CLAIM_COUNT,
    ROUND(AVG(TOTAL_BILLED_AMOUNT), 2) AS AVG_BILLED,

ORDER BY CLAIM_COUNT DESC;

Troubleshooting: The 3 Errors Everyone Hits

1. "Snowpipe Streaming does not support columns of type AUTOINCREMENT, IDENTITY, or columns with a default value"

Your table has DEFAULT UUID_STRING() or DEFAULT CURRENT_TIMESTAMP(). Snowpipe Streaming requires plain columns with no defaults. Generate UUIDs and timestamps in NiFi's ReplaceText instead.

2. "IP X.X.X.X is not allowed to access Snowflake"

SPCS containers use rotating IPs. Don't add individual IPs to your network policy — add a /24 CIDR block (e.g., 153.45.59.0/24).

3. "Value cannot be ingested into Snowflake column of type TIMESTAMP"

Snowpipe Streaming is strict about timestamp formats. Easiest fix: make the column VARCHAR and cast to TIMESTAMP in your Dynamic Tables downstream.
